# Orbitherm Studio ワークベンチ開発ロードマップ v0.1

## 1. 目的

Orbitherm Studio は、FreeCAD 上で熱解析モデルを構築・編集・検証し、外部ソルバーまたは Orbitherm 系ソルバーへ安全に橋渡しするためのワークベンチである。

本ロードマップでは、今後の開発対象を **「熱ソルバーそのもの」ではなく「ワークベンチ機能」** に絞って整理する。  
重点は、以下の3点に置く。

- FreeCAD 上で熱解析モデルを一貫して作成できること
- モデルの整合性・再計算必要箇所・不足設定を可視化できること
- エクスポート前後を含む、実務的なモデル管理環境として完成度を高めること

---

## 2. スコープ

### 本ロードマップの対象
- FreeCAD ワークベンチとしての UI / UX
- モデル作成・編集・検証・表示
- ケース管理
- エクスポート前のモデル整合性確認
- 結果読み戻しと表示の改善
- 内部データ構造の整理
- コマンド群の統合・整理

### 本ロードマップの対象外
- ソルバー本体の高機能化
- 完全な SINDA/ESATAN 文法互換
- 高度な数値計算アルゴリズムの実装競争
- FreeCAD 非依存の CLI ツール化
- 大規模な別製品化

---

## 3. 現状認識

現在の Orbitherm Studio には、README に示されている通り、すでに以下の主要機能群が存在する。

- モデル準備
- 面／ノードのプロパティ設定
- 熱容量計算
- 伝導コンダクタンス計算
- 輻射コンダクタンス計算
- 軌道熱入力
- エクスポート
- ポスト処理・表示

このため、現状は「機能が不足して何もできない」段階ではなく、  
**個別コマンドの集合から、統一された熱解析ワークベンチへ進化させる段階** にある。

今後は新規コマンドを無秩序に増やすよりも、以下を優先する。

- モデル全体の見通しをよくする
- 依存関係を分かりやすくする
- 未設定・矛盾・再計算必要箇所を明示する
- ケース単位で管理できるようにする
- エクスポートをより安全かつ説明可能にする

---

## 4. 開発方針

### 4.1 ワークベンチ中心主義
Orbitherm Studio は、ソルバーの一部ではなく、**熱解析モデルを作るための中心作業環境** として設計する。

### 4.2 コマンド中心からモデル管理中心へ
個別コマンドを順に打つ運用から、  
**「モデル全体の状態を見ながら必要作業を進める」運用** へ移行する。

### 4.3 FreeCAD との自然な統合
FreeCAD の既存 UI / ドキュメント構造 / 表示機構を尊重し、過度に独自流儀へ寄せすぎない。

### 4.4 エクスポート前の品質確保
モデル作成よりもむしろ、**エクスポート時に壊れたモデルを出さないこと** を重要視する。

### 4.5 段階的整理
既存コマンドを捨てて作り直すのではなく、互換を保ちながら徐々に以下へ移行する。

- モデルツリー化
- チェッカー導入
- 統合プロパティ編集
- ケース管理
- エクスポートウィザード化

---

## 5. 目標アーキテクチャ

ワークベンチ内部は、今後できるだけ以下の責務分離を目指す。

### 5.1 Model Layer
純粋な解析モデル定義

- Face/Surface
- Thermal Node
- Material
- Optical Property
- Conductor
- Radiation Coupling
- Heat Source
- Boundary Condition
- Orbit Heat Data
- Case
- Export Setting

### 5.2 Build Layer
FreeCAD 形状から解析モデルを組み立てる処理

- 面抽出
- ノード生成
- 接触候補抽出
- 面属性の継承
- 輻射対象抽出
- 軌道熱入力対応付け

### 5.3 Validation Layer
モデルの不足・矛盾・再計算必要性の判定

- 材料未設定
- 熱物性不足
- 光学物性不足
- 孤立ノード
- リンク欠落
- 再計算必要フラグ
- エクスポート不能条件

### 5.4 UI Layer
ユーザー操作をまとめる層

- ツリー表示
- パネル編集
- 表示切替
- チェック結果表示
- ケース管理
- Export Wizard
- Result Browser

## 6. 優先開発テーマ

---

## Phase 1: 骨格整備

### 6.1 Study / Model ツリーの導入
#### 目的
モデル全体の状態を見える化する。

#### 内容
FreeCAD ドキュメント内に、解析用のまとまりを持つ構造を導入する。

例:
- Study
  - Geometry Set
  - Surface Properties
  - Thermal Nodes
  - Conductors
  - Radiation Links
  - Orbit Heat Cases
  - Export Settings
  - Result Sets

#### 期待効果
- モデル全体の見通しがよくなる
- どこまで作業したか把握しやすくなる
- ケースや結果との対応関係が明確になる

---

### 6.2 Model Checker の実装
#### 目的
未設定・矛盾・再計算必要箇所を明示する。

#### 内容
以下の検査を一覧表示できるチェッカーを導入する。

- 材料未設定面
- 厚さ未設定
- 密度未設定
- 比熱未設定
- α/ε 未設定
- ノード未生成面
- 孤立ノード
- 伝導リンクなし
- 輻射リンク未作成
- HeatSource 設定不足
- Boundary 条件不足
- エクスポート必須項目不足

#### 期待効果
- 作業漏れを減らせる
- エラーを後工程ではなく前工程で潰せる
- ワークベンチとしての完成度が一気に上がる

---

### 6.3 再計算必要フラグの導入
#### 目的
依存関係を見える化する。

#### 内容
たとえば以下のような状態管理を導入する。

- 面プロパティ変更後 → 熱容量再計算必要
- 形状変更後 → ノード再生成必要
- ノード変更後 → 伝導再計算必要
- 輻射対象変更後 → 輻射再計算必要
- 軌道条件変更後 → 軌道熱再適用必要

#### 期待効果
- 古い計算結果のまま進んでしまう事故を防げる
- ユーザーが次に何をすべきか分かる

---

## Phase 2: 編集性向上

### 6.4 統合プロパティ編集パネル
#### 目的
個別編集中心の運用から脱却する。

#### 内容
面・ノード・リンクなどを一括して編集できるパネルを導入する。

対象例:
- 材料
- 厚さ
- 密度
- 比熱
- 吸収率 α
- 放射率 ε
- 熱源値
- 境界条件
- 表示属性

#### 拡張候補
- 複数選択一括編集
- フィルタ編集
- テンプレート適用
- 差分表示
- 初期値復元

#### 期待効果
- モデル作成速度が上がる
- 実務的な大量編集に耐えやすくなる

---

### 6.5 表示切替 UI の統合
#### 目的
単発コマンドを整理し、視認性を上げる。

#### 内容
表示切替系の操作を一つのパネルにまとめる。

例:
- ノード表示 ON/OFF
- ノード番号表示
- 面番号表示
- α/ε 表示
- HeatSource 表示
- コンダクタ表示
- 輻射リンク表示
- 結果温度表示
- Active/Inactive 面の表示切替

#### 期待効果
- 操作が覚えやすくなる
- 説明しやすくなる
- デバッグ効率が上がる

---

### 6.6 手動編集機能の拡充
#### 目的
自動生成後の「最後の詰め」を可能にする。

#### 内容
以下の手動編集を段階的に追加する。

- 手動伝導リンク追加
- 手動放射リンク追加
- 手動熱源追加
- 手動境界条件追加
- ノードの統合 / 分離
- ノード名変更
- グループ化
- 接触候補の採否編集

#### 期待効果
- 実務モデルの微調整が可能になる
- 自動処理だけでは難しいケースにも対応できる

## 9. 開発原則

### 9.1 既存コマンドを活かす
既存コマンドを捨てない。  
まずは UI・状態管理・依存関係管理を足して、既存資産の価値を高める。

### 9.2 小さく導入する
大きな作り直しを避け、1テーマずつ確実に導入する。

### 9.3 可視化を重視する
内部で正しいだけでなく、**ユーザーが理解できること** を重視する。

### 9.4 壊れにくさを優先する
新機能追加よりも、壊れたモデルを出さない・古い結果で誤解しないことを優先する。

### 9.5 研究用途と実務用途の両立
学習・試作に使いやすくしつつ、将来的には実務モデルにも耐えられる構造を目指す。

---

## 10. v0.1 の到達イメージ

v0.1 の到達点は、以下である。

- ユーザーが FreeCAD 上で熱モデル全体を把握できる
- 未設定や矛盾をワークベンチが指摘できる
- どこを再計算すべきか分かる
- ケースごとに条件を整理できる
- エクスポート前に安全確認できる
- 結果をケースと紐付けて管理できる

すなわち、Orbitherm Studio を  
**「コマンド群」から「使える熱解析ワークベンチ」へ進化させる最初の段階** と位置付ける。
